# Feature: WP10.5 — β5 (`clAlphaDot`) schema extension

**Workflow:** feature
**State:** plan (complete)
**Created:** 2026-05-12

## Problem Statement

Phase 1 closed with the phugoid mode undamped (SURFACE-2026-05-11-04) — the airframe diverges in long-window (≥14s) probes under non-zero throttle because β4 (pitch-rate damping) only addresses the short-period mode. arch.md Revision 2026-05-12 D13 settles this architecturally by adopting β5: a per-surface `clAlphaDot` coefficient that augments lift by `clAlphaDot · dα/dt`. This WP lands the schema and the runtime hook with **default 0 → bit-for-bit parity with the existing 246 tests**. Non-zero tuning is explicitly deferred to whichever Phase 2 mission first surfaces the need for sustained level cruise (WP14 waypoint or WP16 combat). Shape mirrors WP6.5 (β1 `incidenceRad`) and WP6.6 (β4 `clQ`) — the established pattern for schema-with-default-zero extensions to `AeroSurface`.

## Plan-time notes (informational; not in tree)

- **No `aircraft.json` changes ship with WP10.5.** β5 is schema-only; the file has no `clAlphaDot` field on any surface, which means every surface defaults to 0, which means bit-for-bit parity with the current Phase 1 behavior. The first non-zero tuning will be inside whichever Phase 2 mission first surfaces a level-cruise need (per WP14/WP16 in wbs.md, and per the arch.md D13 verification approach: ≥30s probe at non-zero throttle).
- **`computeAeroForce` signature change is the one mild ripple.** Optional dt arg keeps existing call sites and test fixtures green. Single new call site in `flightmodel.ts`. Memory `feedback_asymmetric_fix_no_op.md` applies: the augmentation is gated on **both** `clAlphaDot !== 0` **and** `dt !== undefined`, so any future test that builds a surface without explicit `clAlphaDot` sees zero behavior change.
- **prevAoA on AeroSurface is mutable per-tick state.** This is the same pattern as `deflection`. Allocation-free.
- **Risk: a fixture inadvertently sets `clAlphaDot` but doesn't pass `dt`.** Mitigated by the gate above — without dt the augmentation is skipped, so default behavior is preserved. The P1.6 first-tick test exercises this exact case.
- **Risk: dt argument is forgotten at the new flightmodel call site.** Caught by P1.6 "rising α produces positive augmentation" — that test exercises the live path including the flightmodel wrapper if structured to do so. (Alternative: keep tests at the `computeAeroForce` level only and trust manual integration to the flightmodel call site. P1.6 favors the unit-test level; if integration-test coverage is desired, add a flightmodel-level test in P1.8 — flagged as optional.)
- **No backlog conflicts.** SURFACE-2026-05-11-04 (the driver) and the other open SURFACE items (descending-glide, bundle size) are all downstream of or unrelated to this WP.
