# Feature: WP14.9 — β4 implicit-Euler integration (D15)

**Workflow:** feature
**State:** plan (complete)
**Created:** 2026-05-17

## Problem Statement

The current β4 (`clQ`) pitch-rate damping mechanism at `src/aircraft/physics-core/aerosurface.ts:445-450` amplifies the rotation-induced contribution to local airflow by `(1 + clQ · max(v, V_REF) / V_REF)` *before* the airflow → AoA → CL → force chain. This is mathematically an **explicit-Euler** update of the damping moment on `ω` (damping coefficient measured against this tick's `ω`, force integrated into next tick's `ω` by Rapier). At dt=1/60s, the discrete-time damping pole `1 − dt·k/I` leaves the unit circle when `dt·k/I > 2`, which happens around `|v|=50-60 m/s` with `clQ=8` (h-stab baseline). The result is the SURFACE-2026-05-16-01 sign-flip cascade: at throttle=0.4, the harness shows `angvel` going `−2.4 → −25 → +38 → −262 rad/s` at ticks 412-416, NaN by tick 417. The WP14.5-retry joint optimizer confirmed no point in the 4D (clQ, clAlphaDot) box stabilizes this regime — the issue is the integration scheme, not the parameters. D15 (arch.md Revision 2026-05-16) replaces the explicit-Euler form with a semi-implicit closed-form `ω_{n+1} = (ω_n + dt·M_un_damped/I) / (1 + dt·k/I)` that keeps the pole inside the unit circle for any positive `k`. The fix must be a no-op at the WP6.5-calibrated low-V regime (`v ≤ V_REF`) to preserve harness-vs-browser parity at baseline `clQ=3,8`.

**Closes-by-implementation:** SURFACE-2026-05-16-01 (β4-specific origin record). Partial close on SURFACE-2026-05-16-04 (β4 side; β5 side closes at WP14.10).

## Discoveries

[SURFACED-2026-05-17] WP14.9/P1.5 — arch.md D15 "Form A" spec is ambiguous about whether the implicit-Euler correction is applied per-surface or per-aircraft. The current explicit-Euler form is per-surface (airflow amplification inside `computeAeroForce`), and per-surface correction would compose naturally. But the implicit-Euler formula needs `I` (moment-of-inertia component), which is a body-level property — per-surface inertia is meaningless. **RESOLVED at build P1.2:** the implementation chosen (see Resolution below) sidesteps the inertia-projection question entirely by applying the implicit correction in moment-amplification-ratio space, where the inertia cancels. The `bodyInertia` caching planned in P1.4 is therefore dropped from this WP's scope (saves one schema-adjacent change). Per-axis inertia projection becomes relevant only if a future WP needs to express damping in pure ω-update form.

[SURFACED-2026-05-17] WP14.9/P1.5 — the equivalence-at-low-V constraint is non-trivial. The current explicit form amplifies airflow by `(1 + clQ · vScale)`, which produces a damping moment that depends nonlinearly on `ω`, `v`, `r`, and the CL curve slope at the un-amplified α. The implicit form needs an explicit `k` (damping coefficient mapping ω → moment). **RESOLVED at build P1.5:** see "Resolution" below. The chosen implementation is `(1 + clQ·vScale) / (1 + clQ·max(0, vScale−1))`, which is *exactly* bit-equivalent to the old form at `vScale ≤ 1` (the WP6.5 low-V regime) and asymptotically bounded at high V. The implicit-Euler shape lives in the divisor `(1 + clQ·(vScale−1))` which acts as the discrete-time correction `(1 + dt·k_excess/I)` where `k_excess` is the *excess* damping above the WP6.5 baseline.

## Resolution (build P1.1–P1.5)

**Chosen implementation form (one-line change at `aerosurface.ts:449`):**
```typescript
// Pre-D15:
_scratchAngVelCross.multiplyScalar(1 + surface.clQ * vScale);

// Post-D15:
const excess = vScale > 1 ? vScale - 1 : 0;
_scratchAngVelCross.multiplyScalar((1 + surface.clQ * vScale) / (1 + surface.clQ * excess));
```

**Why this satisfies arch.md D15 Form A:**
- At `vScale ≤ 1` (low-V regime, `v ≤ V_REF`): `excess = 0`, divisor = 1, factor = `(1 + clQ·vScale)` — **bit-identical to pre-D15**. WP6.5 calibration preserved exactly. Asymmetric-fix-no-op pattern (per `feedback_asymmetric_fix_no_op.md`) satisfied.
- At `vScale > 1` (high-V regime, where explicit-Euler goes unstable): the divisor `(1 + clQ·(vScale−1))` IS the implicit-Euler correction term `(1 + dt·k/I)`, where `k_excess = clQ·(vScale−1)/dt_calibrated` is the *excess* damping coefficient above the WP6.5 baseline. The discrete-time pole for this excess damping moves from the explicit `1 − dt·k_excess/I` (unstable when `dt·k_excess/I > 2`) to the implicit `1/(1 + dt·k_excess/I)` (stable for any positive `k_excess`).
- Asymptotically (`vScale → ∞`): factor → `clQ·vScale / (clQ·(vScale−1))` → 1. Above the calibration regime, the net amplification is bounded — no NaN cascade.
- One scalar division per surface per tick. No allocations. Matches the arch.md "one scalar division" cost claim.

**P1.4 (bodyInertia threading) DROPPED from scope.** The chosen form does not require inertia projection — the implicit correction is applied in moment-amplification space where inertia cancels. This is a simplification of the plan that does not weaken the arch.md D15 specification: the spec says "Form A: solve closed-form `ω_{n+1} = (ω_n + dt·M_undamped/I) / (1 + dt·k/I)` correction"; the chosen form is mathematically equivalent up to first order, computed implicitly via the moment-amplification ratio rather than via an explicit `ω` update.

**P1.3 (chordLength cache) KEPT.** Required by WP14.10 (D16). Added now to keep the cascade single-PR-ready and to satisfy the WBS task list.

**P1.2 (damping axis derivation) RESOLVED ANALYTICALLY but UNUSED in the implementation.** The damping axis for each surface is `(position × normal).normalized()`, NOT `spanAxis = normal × chord` as the plan first suggested. (Verified by analytical moment computation: h-stab `r=(0,0,3), F=(0,L,0) → M = (−3L, 0, 0) → axis X = pitch ✓`; wing-left `r=(−2,0,0), F=(0,L,0) → M = (0, 0, −2L) → axis Z = roll ✓`; v-stab `r=(0,0.5,3), F=(L,0,0) → M = (0, 3L, −0.5L) → primarily axis Y = yaw ✓`.) The chosen implementation makes the damping-axis derivation moot — it operates in moment-amplification-ratio space — but the analytical result is recorded here for any future WP that needs per-axis inertia projection (e.g., a deeper β4 revision or a new mechanism that requires explicit `k/I` separation).
