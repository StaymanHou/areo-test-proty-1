# Feature: WP16 — Combat Mission

## Problem Statement

Phase 2's exit criteria require all four mission types from the vision to be minimally playable. Free flight (WP13), waypoint patrol (WP14), and takeoff/landing (WP15) are shipped. WP16 — **combat** — is the last and the largest remaining Phase 2 risk (R6 in research). The player needs a mission where the win condition is "destroy a hostile target" and the fail condition is "you crashed (or were destroyed)." This requires four new subsystems wired into existing scaffolding: a forward-firing weapon, projectile lifecycle, target entity with hitpoints, and an `combat-ai` script hook that ties them all to the mission runner per arch.md D11.

Existing scaffolding ready for WP16:
- `Mission.scriptHook?: string` already in `types.ts` and validated by `parse.ts`.
- `MissionRunner` already fires the hook BEFORE objective evaluation each tick (deliberate ordering documented in arch.md D11 + runner.ts line 109-114), so a hook can flip `destroy-target` `ObjectiveState.completed` synchronously.
- `HookFn` signature is allocation-free per tick (`HookState`, `AircraftState`, `readonly ObjectiveState[]`, `dt`) — already designed for the combat case.
- `registerHook(name, fn)` in `src/mission/hooks/registry.ts` throws on duplicate registration; getHook lookup is O(1) Map.
- `destroy-target` is a first-class `ObjectiveKind` in `MISSION_TYPES` / `OBJECTIVE_KINDS` / `Objective` union.
- `parse.ts` already validates `DestroyTargetObjective` shape (`{kind: 'destroy-target', targetId: string}`).

The combat mission JSON wire-up is therefore additive: one new JSON in `public/missions/combat.json`, one new manifest entry in `public/missions/index.json`, one `registerHook('combat-ai', …)` call from `main.ts`, one new `src/mission/hooks/combat-ai.ts` with the hook implementation. The new subsystems live behind the hook — they are NOT new public APIs into the mission framework. This contains the risk per R6.

Problem statement unchanged — the back-loop from Phase 4 verify-self is for a missing debug accessor (`window.__combat.getReturnFireSnapshot` + `getPlayerHp`), not a root-cause shift. The fail-by-HP system is correct (proven by unit tests); we need its state observable from the live system to complete verify-self. [Updated 2026-06-07: scoped re-entry, no root-cause change.]

## User Stories

- **As a casual gamer**, I pick "Combat" from mission select, spawn at V_trim altitude, see a hostile target somewhere ahead in the world, fire my gun by holding spacebar, watch the target take hits and explode, and win the mission.
- **As a casual gamer**, if I crash my aircraft or get shot down by the target's gun first, I see the standard fail banner and return to mission select.
- **As a developer adding more Phase 2/3 content**, the combat-ai script hook is one isolated module with no special-case branches inside `MissionRunner` or `flightmodel.ts` — the hook gets per-tick `(state, aircraft, objectives, dt)` and does its own thing, exactly like the D11 contract specifies.

## Acceptance Criteria

1. **Mission JSON loads and runs.** `public/missions/combat.json` exists with `type: 'combat'`, `scriptHook: 'combat-ai'`, a single `destroy-target` objective, and the V_trim=78 spawn convention (per CLAUDE.md Rule #9). It appears in `public/missions/index.json`. Deep-link `localhost:5173/?mission=combat` boots straight into the mission. `parse.ts` validates the JSON without errors.
2. **Player gun fires.** Holding Space (a new control) emits a forward-firing gun projectile at a tunable rate-of-fire (target ~5 rounds/sec). Projectiles spawn from the aircraft body origin, travel along the aircraft's forward axis at a high relative velocity (target ~600 m/s muzzle velocity), and despawn at a fixed range (target 1500 m). Allocation-free per tick — projectiles are managed by a fixed-size pool.
3. **Target is a stationary ground target.** A single visible enemy object at a fixed world position (~600 m forward of spawn at ground level — within the V_trim turn radius for a casual-flyable approach). Has hitpoints (target 5). Has a Three.js mesh + a Rapier collider (or a math-only AABB) so projectile-vs-target hit detection works. NOT an AI-flown aircraft for v1.
4. **Hit detection.** Projectile path vs target collider produces a hit event that decrements the target's hitpoint counter by 1 per hit. On `hp ≤ 0`, the target is marked destroyed (mesh removed or visibly switched to "destroyed" state) and the `destroy-target` objective's `ObjectiveState.completed = true`. Within the same tick, the runner's objective-evaluation pass observes the flag and the mission transitions to `won` per the existing `all-objectives` win path.
5. **Damage model for the player.** The player aircraft has a hitpoint counter (target 10). The target retaliates: a simple "the target shoots back" mechanism (one return-fire projectile every ~1 sec aimed at the player's current position with a ~30 m/s lead — easy to dodge by maneuvering). On player `hp ≤ 0`, the runner's existing `failCondition === 'crash'` path is triggered (or a new path is added — TBD at plan time; see Open Questions). Player crash-into-ground continues to use the existing `failCondition: 'crash'` path.
6. **Combat HUD overlay.** The existing DOM HUD (`src/hud/dom-hud.ts`) shows player HP (e.g. "HP: 10/10") and the target's HP (e.g. "Target HP: 5/5") during the combat mission. The objective string ("Destroy the target") comes from the existing `formatActiveObjective` path; only the HP readouts are net-new.
7. **No regressions in the other three missions.** Vitest unit suite green (currently 643/643); Playwright `casual-flight.spec.ts` smoke + the four-mission e2e checks pass. The combat hook registration is gated so it only fires for the combat mission (the hook registry only invokes registered hooks named by `mission.scriptHook` — already-enforced by `MissionRunner.start`).
8. **Scripted-input harness probe shape.** A new `tests/e2e/combat.spec.ts` (or extension to an existing combat smoke) drives the combat mission via `?script=hold:Throttle=…@0:N,hold:Space@…` and asserts the mission transitions to `won` within a deterministic time window. Per CLAUDE.md "Browser-walkthrough discipline," ANY agent-side time-sensitive observation of combat goes through `?script=` not `page.keyboard.press`.

## Out of Scope

- **Missile / guided weapons.** v1 picks gun-only. Lock-on, lead indicators, missile guidance, IR/radar models — explicitly deferred. Adds aiming-assist surface that conflicts with the "casual gamer" target audience.
- **AI-flown enemy aircraft.** Stationary ground target only. AI flight (reuse flight model + dumb pursuit/evade FSM) is a richer-gameplay path but adds an entity-management surface (multiple aircraft, multiple Rapier bodies, multi-aircraft camera/HUD assumptions) — defer to Phase 3 polish or a follow-on combat mission.
- **Multiple weapons or weapon selection.** One gun, one trigger button, one ammo behavior. Loadout selection is Phase 3 polish at the earliest.
- **Damage states beyond destroyed.** Component damage (wing torn off, engine sputters, control surface jammed) is out of scope. Single hitpoint pool, binary destroyed/alive state.
- **Sound effects.** Per CLAUDE.md "Phase discipline" — audio is WP19 (Phase 3). The combat mission may emit `console.debug` events for hit/destroy that a future audio pass listens to, but no Web Audio in WP16.
- **Particle effects (muzzle flash, projectile trail, explosion).** Per CLAUDE.md "Phase discipline" — visual polish is WP20. WP16 ships simple debug visuals (a small line / sphere for the projectile, a "destroyed" state change for the target) that WP20 replaces.
- **AI behavior tree / FSM internals.** Deferred — there is no AI aircraft in WP16. The "target shoots back" mechanism is the simplest possible per-tick "spawn return-fire at fixed cadence with simple aim" routine; it lives in `combat-ai.ts` without a state machine.
- **WP17 (Phase 2 verification) bundling.** WP17 is the next critical-path WP after this one; WP16 ships ONLY the combat mission + its subsystems. WP17 does the cross-mission integration sweep.
- **Cessna vs jet airframe choice for the combat mission.** Recommend Cessna default (matches existing four-mission Cessna-default set per the resume note). Per-mission `config?:` already exists from WP14.20 — if operator playtest reveals Cessna is too slow for the gun-vs-stationary-target gameplay, the mission JSON can carry `config: "mig15"` as a one-line change without architectural impact. Pinned at plan time.

## Technical Constraints

- **No new arch.md decisions required.** D11 already defines the mission framework + scriptHook contract. The internal architecture of `combat-ai.ts` (a single per-tick function with private state) is explicitly a WP16-internal decision per arch.md line 50 ("internal AI architecture (behavior tree vs FSM) is a WP16-internal decision; arch.md does not pre-commit"). Since there is no AI aircraft in WP16, no state machine is needed at all — the hook is straight imperative code.
- **Allocation-free per-tick discipline.** `HookFn` runs inside the physics tick (60 Hz fixed). Projectile pool and target state must be pre-allocated. Use simple typed arrays / preallocated object pools, not `new Projectile()` per shot. Pattern is already established in `flightmodel.ts` / `aerosurface.ts` (reused `Vector3` buffers).
- **Right-handed Y-up, no axis-flip at boundaries.** Per arch.md D7. Projectile forward direction = aircraft body's `-Z` (matches mission JSON spawn convention `linvel.z: -78`). Target placement coordinates use the same frame as `mission.spawn.position`.
- **Mission-runner contract preserved.** The runner's `tick(aircraft, dt)` is the only entrypoint into combat logic. The hook does NOT touch `MissionRunner`'s internals; it mutates `ObjectiveState.completed` via the array reference the runner hands it, and it mutates its own `HookState` bag for projectile-pool + target state. (This is the existing contract — confirmed at runner.ts line 109-114 and registry.ts line 25-30.)
- **CLAUDE.md Browser-walkthrough discipline.** Any agent-side automated verify-self of combat MUST use `?script=` not `page.keyboard.press`. The harness already supports `hold:Space@…` via the `KeyCode` grammar.
- **`MissionRunner` does NOT yet expose a way for the hook to fail the mission.** Currently the runner's fail evaluation only sees `failCondition` (crash / timeout / out-of-bounds). If "player HP reaches 0" needs to fail the mission, two options exist: (a) hook signals fail by mutating a runner-readable bag (small runner change) OR (b) hook simulates a "crash" by setting aircraft.position.y = -1 + a large |linvel.y| (gross hack; rejected). This is the only `MissionRunner` API extension this WP may need — see Open Questions.
- **HUD interface needs a "secondary status string" or "combat HP readout" surface.** Current `HUD` interface (`src/hud/HUD.ts`) supports `setObjective` + `setStatus`. Combat HP needs a new method or repurposing of an existing one — see Open Questions.

## Open Questions — resolved at plan time

All five resolved here. Recorded for future reference; no further research needed.

- **Q1 RESOLVED — Player-fail-by-HP:** Extend `MissionRunner` with a minimal API. Add `setHookFailFlag(reason?: string)` and check it in the fail evaluation pass (between the existing `crash` / `timeout` / `out-of-bounds` branches). The hook calls `runner.setHookFailFlag('shot down')` when player HP hits 0. Cost: ~10 lines in `runner.ts` + 1 unit test. Rejected alternative: hook fakes a crash by mutating aircraft position — leaks combat state into physics layer, untestable.
- **Q2 RESOLVED — Combat HP HUD:** Add `setCombatHP(playerHP: number | null, targetHP: number | null)` to the `HUD` interface; both `null` hides the readout (default for non-combat missions). `DomHud` adds two `<div>` rows under the existing objective row. Rejected alternative: stuffing HP into the objective string — conflates concerns and breaks the existing test that asserts objective rewrites.
- **Q3 RESOLVED — Projectile rendering:** Pre-allocated pool of 32 small Three.js `Mesh`es (each a 0.5m yellow sphere via `SphereGeometry` reused per pool slot). `mesh.visible = false` when inactive, `visible = true` + `position.copy(...)` when fired. Per-tick render-frame sync from the projectile pool's plain-data array. No InstancedMesh, no Lines — WP20 swap point.
- **Q4 RESOLVED — Target visualization:** Single hazard-orange Three.js `Mesh`: `BoxGeometry(8, 4, 8)` with `MeshLambertMaterial({color: 0xff6a00})`, placed at `(0, 2, -600)` (centered, 4m above ground, 600m forward of V_trim spawn). On destruction, switch material color to dark grey + rotate 90° about Z to convey "fell over." WP20 polish target.
- **Q5 RESOLVED — Hit detection:** Per-tick AABB sweep in TS inside `combat-ai.ts`. Each tick, for each active projectile, check whether its current position lies within the target's static AABB (`±halfExtents` from target center). O(N×1) where N ≤ 32 — trivially cheap. Same AABB pattern checks return-fire-projectile vs player (player AABB = ±2m from `aircraft.position`). Rejected: Rapier collision events — adds Rapier-body overhead per projectile + couples lifetime to physics-tick callbacks for a v1 feature that doesn't need physics-accurate hits.

### Note on Phase 2 design deviation from plan

The plan said `Projectile pool ... in combat-ai.ts HookState`. At impl time, I moved the projectile state from the runner-owned `HookState` bag to a module-level singleton in `combat-ai.ts`. Reason: render-loop (`onRender`) and debug accessors (`window.__combat.*`) need a stable reference to the projectile array, and routing through `HookState` would require exposing the runner's private state via a new public API. The module-level singleton with `getCombatState()` accessor keeps the runner contract clean. Per-mission reset is explicit via `resetCombatState()` called from `main.ts`'s `startMission`. This is a minor implementation-detail variance from the plan's wording but preserves the spec's allocation-free + per-mission-reset semantics — not a back-loop to plan.
