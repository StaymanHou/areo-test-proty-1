## Problem Statement

WP14.5's hand-tuning of β5 (`clAlphaDot`) burned the 3-attempt budget on three sparse points and produced no usable signal in a continuous 2-knob × 3-regime space. The operator escalated to a major detour: build a Node-side headless physics harness that runs at many multiples of wall-clock, scored against an envelope-probing fitness function, driven by a gradient-free optimizer. arch.md Revision 2026-05-12 (afternoon) landed that decision as D14 with a binding cascade — WP14.6 → WP14.7 → WP14.8 → rescoped WP14.5. **This WP (14.6) is the first step:** make the shipped flight-model code callable outside the browser (file-move reorg into `src/aircraft/physics-core/`) and stand up a harness↔browser **parity test** that asserts bit-identical trajectories (`|Δ|<1e-6` per scalar) across the WASM boundary. At this WP the harness side is a Vitest synthetic stub — pure-TS loop calling the same `physics-core/` modules — which proves the modules are framework-agnostic-callable. WP14.7 will replace the stub with the real Node Rapier-WASM harness.

**Plan-time discovery worth surfacing now (NOT a back-loop, just a refinement of the split criterion):** Every existing file in `src/aircraft/` imports `three`, but `aerosurface.ts`, `config.ts`, `flightmodel.ts`, and `state.ts` import only `Vector3` / `Quaternion` (pure math classes that work in Node — Three exports them with no browser-API dependency). Only `rigidbody.ts` imports rendering primitives (`Scene`, `Mesh`, `BoxGeometry`, `MeshStandardMaterial`, `Group`). The arch.md split rule said "does the file import `three`?" — applied literally, that puts *nothing* under `physics-core/`. The arch *intent* is "does this code require browser APIs to run?", and only `rigidbody.ts`'s mesh-building does. **The plan adopts the intent-rule**: math-only `three` imports (Vector3, Quaternion) are allowed under `physics-core/` and remain in the same `three` package they already use. Rendering primitives (Scene, Mesh, geometries, materials) are the actual browser-coupling and stay in `src/aircraft/rigidbody.ts`. CONVENTIONS.md will document this distinction. This refinement is consistent with the arch revision; no back-loop needed.

**3rd-party probe check:** N/A — no new external APIs. Three.js (Vector3/Quaternion only) and Rapier-WASM are both already integrated and have multi-WP track record. The harness in WP14.7 will use `@dimforge/rapier3d-compat` in Node, which is the same package the browser uses — no new 3rd-party. WP14.6 doesn't run the harness yet.

## Test Triage — step.test.ts "advances the world by exactly one tick and sets world.timestep to dt"
Classification: Obsolete test — over-specified precision in a newly-authored test (Phase 1 verify-codify). The assertion `toBeCloseTo(1/60, 10)` expected ~5e-11 tolerance; Rapier's `world.timestep` setter round-trips through f32 producing ~8.7e-10 error.
Confidence: high
Evidence: Vitest output shows `expected 0.01666666753590107 to be close to 0.016666666666666666, received difference is 8.692344032057431e-10` — f32 bit-pattern truncation of f64 `1/60`. Production sequence (`world.timestep = dt; world.step()` in `src/main.ts:86-87`) already does this; no real-world correctness implication. The other 3 step.test.ts cases pass with looser precision and exercise the actual contract (one-tick advancement, thrust application, determinism vs. inline path).
Action: Loosen the assertion to `toBeCloseTo(dt, 6)` (~5e-7 tolerance, comfortably above f32 round-trip). No production code change.

## Test Triage — trajectory-buffer.test.ts "round-trips a single row exactly (lossless for the numeric fields)"
Classification: Obsolete test — over-specified deep-equal in newly-authored Phase 2 test. `Euler.setFromQuaternion(identityQuat)` produces `pitch: -0`; `csvToTrajectory` parses `"0"` → `+0`. `-0 === +0` is true in JS but `toEqual` distinguishes them in deep-equal mode.
Confidence: high
Evidence: Vitest diff shows `pitch: -0` (expected, from buffer's Euler) vs `pitch: 0` (received, from CSV round-trip). All 11 other trajectory-buffer cases pass; this is a JS representation detail with zero behavioral impact. CSV serialization losing the negative-zero sign is correct behavior.
Action: rewrite the round-trip assertion to compare with numeric-equality semantics (per-field `toBe` or `toBeCloseTo`, which treat ±0 as equal). No production code change.

## Retrospect

- **What changed in our understanding:** the arch.md split criterion "does the file import `three`?" was literally wrong — every `src/aircraft/` file imports `three`, but only `rigidbody.ts` imports rendering primitives (Scene/Mesh/Geometry/Material). The intent-based rule ("requires a browser API to run") is what the harness actually needs. Operator was asked before any file move and approved adopting the intent rule + documenting it in CONVENTIONS.md. No arch.md back-loop was needed because the *intent* is unchanged; only the mechanical test refined.
- **Assumptions that held:** (a) `Vector3`/`Quaternion` math classes work in Node — confirmed when the Vitest synthetic stub instantiated them with no special setup. (b) Rapier-WASM is bit-deterministic across the same binary in browser and Node — confirmed by 1800-tick bit-identity (`|Δ|<1e-6`) once the `Aircraft.reset` bug was fixed. (c) Plan-time grep would catch all importers — *almost* held; missed `src/mission/hooks/registry.ts` (one level deeper), caught by `tsc --noEmit` immediately. Cheap recovery; bounded cost.
- **Assumptions that were wrong:** the biggest surprise was `Aircraft.reset()` leaving stale Rapier force/torque accumulators. Plan-time problem statement said "no behavioral change intended" for Phase 1; in Phase 2 the parity test caught a real production bug that had been silent in WP6.5–WP14 because no mission's first tick had previously been observed at < 1e-6 resolution. The bug would have manifested on mission restart (return-to-menu → new mission) — the next mission's tick 0 would inherit forces from the previous mission's final tick. Now permanently regression-anchored by `tests/parity-diff.test.ts`.
- **Approach delta:** plan had `FlightModel` consume `Aircraft`; build widened the constructor to consume the lower-level `AircraftBody` (so the harness doesn't need the Three.js wrapper). This wasn't in the plan but is a natural follow-on of the `Aircraft extends AircraftBody` refactor and required for the synthetic stub. No scope creep — the change was tiny (one type widening, the existing `Aircraft.config`/`Aircraft.body` properties live on the base class).
- **Test triage:** two over-precise newly-authored assertions caught during their first run — `step.test.ts` over-specified f32 precision (tightened from `toBeCloseTo(dt, 10)` to `toBeCloseTo(dt, 6)`), and `trajectory-buffer.test.ts` deep-equal hit `±0` mismatch from `Euler.setFromQuaternion(identity)` (rewrote per-field with `===` to treat `±0` as equal). Both classified obsolete-test (high confidence), both documented in this file's `## Test Triage` section, no production code involved.
- **Cascade impact:** WP14.7 inherits a working parity test, a single-fixture seed for the fixture-loop pattern, and a clean physics-core/ namespace. WP14.7's plan should consolidate the synthetic-stub's world-construction (ground + tower colliders) into a shared "world fixture" helper that both `tests/parity-diff.test.ts` and `tools/tune/harness.ts` consume — a comment in `parity-diff.test.ts:50-56` already nudges this. No SURFACE-to-backlog needed.
