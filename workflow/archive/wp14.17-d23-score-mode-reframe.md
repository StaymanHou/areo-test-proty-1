## Problem Statement

The D14/D17/D18/D19/D20/D21/D22 cascade reached a qualitative breakthrough at WP14.16 (all 3 regimes c0 PASS, deployed −93.2M, 22× better than WP14.15) but residual c2 envelope penalty −93M from observed chaotic non-linear motion (AS ±84 m/s, pitch saturated ±90°) revealed a deeper score-function abstraction gap: D21's per-regime equal-weight assumption that "every throttle = level cruise" doesn't match airframe physics. A real Cessna at idle DESCENDS by design; at low/mid throttles under WP14.16 drag, **no level-flight equilibrium exists at any AS** (D_min=1959N >> T at τ=0.05/0.15). The WP14.16 c0 PASS at low/mid was a 1-second-window artifact (identical to WP14.15's high-regime artifact / D22 finding). D23-γ-evolved reframes the score function to dispatch per regime: `low=controlled-descent` (sink rate [1,5] m/s, pitch within [-30°, +10°]); `mid=slow-flight-or-shallow-descent` (AS within [25,50], sink rate [-1,3], pitch ±20°); `high=level-cruise` (preserves D21's level-flight check + AS_ENVELOPE around target 85, adds optional T=D regime guard at tick ≥600). WP14.17 implements the dispatch + 7 binding Vitest cases + back-compat (legacy callers without `regimeMode` fall back to D21 level-cruise unchanged). Tooling-only: no `aircraft.json` change, no mechanism additions, no bound changes. Closes-by-impl at WP14.18 Branch A together with the 12-SURFACE chain (13 SURFACEs total).

## Plan-time notes (not part of Work Tree)

**Why single-phase:** WP14.17 is tooling-only, ≤200 lines of code additions, all in a single file (`tools/tune/score.ts`) with a single test file extension (`tools/tune/score.test.ts`). Splitting into multiple phases would add overhead without value.

**Back-compat strategy:** every new field on `ScoreEnvelopes` is optional (`?:`). The default dispatch is `level-cruise` when `regimeMode` is absent OR when a regime label doesn't appear in `regimeMode`. The existing `regimeScore` is refactored (not deleted) into `levelCruiseScore`; the new `regimeScore` is a thin dispatcher. This preserves bit-identical output for any caller not opting into D23.

**Why airframe constants in envelopes (P1.4-sub):** the T=D guard needs to compute drag at each tick. Two options:
1. Import aircraft.json into score.ts — couples score.ts to FS reads, breaks pure-compute property.
2. Pass airframe constants through envelopes (chosen) — keeps score.ts pure-compute, makes T=D testable without aircraft.json.

The chosen approach also lets the optimizer skip the T=D guard (just don't populate `airframe` in envelopes) — useful for fast iteration where mass/thrust aren't varying.
