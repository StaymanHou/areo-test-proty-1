# Feature: WP14.7 — Node Rapier-WASM harness (single-probe driver)

## Problem Statement

WP14.6 shipped the framework-agnostic `src/aircraft/physics-core/` namespace and a parity test that proves a synthetic Node-side stub (in-process Vitest, no Three.js) matches the browser trajectory bit-identically (|Δ|<1e-6 over 1800 ticks). WP14.7 promotes that stub into a real CLI driver — `tools/tune/harness.ts` — that boots Rapier-WASM in Node, loads `public/config/aircraft.json`, takes initial conditions + parameter overrides + tick count from CLI args, steps `physics-core/step.ts` in a tight loop, and emits a deterministic trajectory CSV. The harness becomes the inner loop the WP14.8 Nelder-Mead optimizer will call repeatedly. Acceptance gate is **parity** — the WP14.6 `tests/parity-diff.test.ts` must keep passing when its Node-side trajectory comes from the harness instead of the synthetic stub. Two carry-forwards from WP14.6: (a) **world-fixture helper** consolidating ground + tower collider construction from `src/main.ts`, `src/world/{terrain,landmarks}.ts`, and `tests/parity-diff.test.ts:50-56` into a single shared module (the parity test's inlined values currently *diverge* from `FlatTerrain.getColliderDesc()` — `(2000, 0.001, 2000) @ (0,0,0)` vs `(2000, 0.1, 2000) @ (0,-0.1,0)` — passing only because the aircraft spawns at y=50 and never touches ground in the 30s fixture; this is a latent bug for any future ground-interacting fixture); (b) **envelope fixtures** — extend `PARITY_FIXTURES` to include `throttle-low` (0.05) and `throttle-high` (0.4), closing the `feedback_verify_self_envelope.md` gap that WP14.6 explicitly deferred.
