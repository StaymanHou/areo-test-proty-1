### SURFACE-2026-05-24-09 — CRITICAL: Rapier per-tick force accumulator never cleared; `body.resetForces()` + `body.resetTorques()` missing from `src/aircraft/physics-core/step.ts:32-36`; forces compound multiplicatively (at tick n applied force = (n+1) × intended); this is the root cause of the entire D14→D23 cascade — 8 architect revisions chased symptoms of a one-line integrator bug
- **Source:** task:act (WP14.18 investigation spike, 2026-05-24)
- **Target level:** product:arch (D24 cycle driver — supersedes SURFACE-2026-05-24-08 as the actionable architecture concern; the D23 cascade's mechanism stack was tuning under a broken integrator)
- **Type:** bug / integrator-correctness / cascade-root-cause
- **Priority:** high (this is the cascade root cause; gates WP15/WP16/WP17 Phase 2 mission content; resolves-on-fix the chain of pretty much everything the cascade has filed since SURFACE-2026-05-12-01 in some form)
- **Summary:** The Node-side harness (`tools/tune/harness.ts` via `src/aircraft/physics-core/step.ts`) and presumably the browser game loop never clear Rapier's per-tick force accumulator between physics steps. `body.addForce()` and `body.addForceAtPoint()` accumulate into a buffer; `world.step()` integrates but does NOT auto-clear the buffer. The contract (per Rapier docs and the comment at `src/aircraft/physics-core/flightmodel.test.ts:411`: "`resetForces` clears it before world.step") requires user-side `body.resetForces(true); body.resetTorques(true);` before each `applyForces()` call. The production path omits this. Result: at tick n, the cumulative force applied during integration is **(n+1) × intended force**.
- **Evidence (probes at tools/tune/probe-*.mjs):**
  - `probe-thrust-only.mjs` — high-regime fixture (throttle=0.4, init AS=30), aero zeroed (areas=1e-6). Observed Δvz per tick: tick 0 = -0.040 ✓ (expected per F=ma at thrust 2400 N / mass 1000 kg / 60 Hz); tick 1 = -0.080 (2× expected); tick 10 = -0.440 (11×); tick 55 = -2.240 (56×). **Δvz at tick n = -(n+1) · 0.040 m/s** — exactly consistent with force accumulator compounding.
  - `probe-noeverything.mjs` — throttle=0 + aero zeroed → vZ correctly stays at -30, vY correctly accelerates to -8.83 m/s under gravity at t=0.9s. Rules out integrator instability or any non-force-accumulator effect; isolates the bug to user-applied forces.
  - `probe-fix-test.mjs` — directly drives the physics step from a script, adding `aircraft.body.resetForces(true); aircraft.body.resetTorques(true);` before each `applyForces()` call. **Result: Δvz per tick = -0.0400 exactly at every tick.** Fix confirmed.
- **Buggy code site:** `src/aircraft/physics-core/step.ts:32-36`. Browser game loop in `src/main.ts` (or wherever the per-tick equivalent runs) almost certainly has the same omission since the WP14.7 parity test passes (it would not pass if browser and harness behaved differently).
- **Fix (one-line, two-call):**
  ```typescript
  // src/aircraft/physics-core/step.ts
  export function step(world, _aircraft, flightModel, inputs, dt): void {
    _aircraft.body.resetForces(true);
    _aircraft.body.resetTorques(true);
    flightModel.applyForces(inputs.throttle, dt);
    world.timestep = dt;
    world.step();
  }
  ```
- **Reproduction recipe:**
  ```
  npx tsx tools/tune/probe-thrust-only.mjs  # shows compounding bug
  npx tsx tools/tune/probe-fix-test.mjs     # shows fix works
  ```

### SURFACE-2026-05-17-02 — arch.md D17 dampAxis cross-product order is sign-inverted vs textbook damping convention
- **Source:** feature:build (WP14.9b Phase 1, 2026-05-17)
- **Target level:** product:arch (D17 errata — single-character fix to literal spec text)
- **Type:** arch-errata / sign-convention
- **Priority:** medium (cosmetic — implementation already proceeds with the corrected sign per CLAUDE.md Rule #1 live-derivation; arch.md prose just needs to match what the code does)
- **Summary:** arch.md D17 (Revision 2026-05-17) and the WP14.9b WBS entry both specify `dampAxis = (position × normal).normalized()`. By right-hand rule on the canonical h-stab (`position=(0,0,3), normal=(0,1,0)`), this gives `(−1, 0, 0)` = anti-pitch axis. After flipping the cross-product order in code, behavior aligns with the test's expectation.
